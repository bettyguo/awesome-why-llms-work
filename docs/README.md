# `docs/` — the GitHub Pages site

> Source for the live site at <https://bettyguo.github.io/awesome-why-llms-work/>.

The site is deliberately small: vanilla HTML/CSS/JS + Tailwind via CDN, no
build step. It pulls structured content from `docs/data/ledger.json`, which
is generated by `scripts/extract_ledger_json.py` from the canonical
`programmes/*.md` files. Truth lives in the Markdown; the site is a view.

## Pages

| File | What |
|------|------|
| [`index.html`](index.html) | Landing — hero, programme cards, taxonomy diagram, CTAs. |
| [`ledger.html`](ledger.html) | Filterable, searchable table of all 41 tracked claims. |
| [`superposition-demo.html`](superposition-demo.html) | In-browser toy-model-of-superposition (Elhage et al. 2022) — trains a 2-layer net on sparse synthetic features and visualizes the d=2 bottleneck columns. |

## Local dev

Pages are static and `docs/` is self-contained. Any local web server works:

```bash
python -m http.server -d docs 8000
# then visit http://localhost:8000/
```

> All assets the HTML references (`hero.png`, `taxonomy.svg`, etc.) live
> inside `docs/`, so the directory is browseable locally with no path
> rewriting. The deploy workflow regenerates fresh copies during build
> from the canonical files at the repo root.

## Deploy modes

Either GitHub Pages mode works because `docs/` is self-contained:

- **Source: GitHub Actions** — `.github/workflows/pages.yml` regenerates
  `data/ledger.json` and the diagrams, then uploads `docs/` as the
  Pages artifact.
- **Source: Deploy from a branch → `main` → `/docs`** — Pages serves
  `docs/` directly with no workflow involved. Useful as a fallback if
  the Actions-based deploy is misconfigured.

## Updating the data

```bash
# After editing any programmes/*.md ledger row:
python scripts/extract_ledger_json.py
# Regenerate flat papers index:
python scripts/generate_papers_index.py
# Regenerate the diagrams:
python scripts/render_taxonomy.py
python scripts/render_hero.py
```

`scripts/extract_ledger_json.py` reads the per-claim tables under `## 5.
Falsification Status — by claim` in each programme file and dumps them to
`docs/data/ledger.json`. The site's JS (`js/main.js`, `js/ledger.js`)
fetches that JSON at load.

## Why no framework / no build?

- The audience is researchers, not designers; speed-of-loading and
  source-readability beat framework ergonomics.
- The site is read-only; it does not need state management.
- A `<script src="https://cdn.tailwindcss.com">` plus 4 short JS files
  is enough for the experience. Adding a bundler would 5× the maintenance
  cost.

If the site outgrows this, the time to migrate to a framework is when we
need *server-side rendering* of programme files (Markdown → HTML), which is
not yet — programme files are linked back to the repo for serious reading.
